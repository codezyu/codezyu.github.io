data block组装成page的方式

假设这样的场景，如果给你大小不一的data block，由于SSD的写入是4KB，那么应该如何组装数据块？
对于Linux而言，做了页的抽象，意味着，内存中的页其实跟SSD中的页是一致的，因此它不需要考虑如何组block的问题，因为你的写入操作是对某个页上面进行操作。对于操作系统而言，跨页是由应用维护的问题
对于数据库而言，采用的则很简单，当前页装不下了，则换一个页，避免数据跨页（大数据块除外）

  • 对于索引页则会不一样，原因在于会触发分裂，保证两个页的数据尽量相等
    所以正常场景下，一般直接无脑append即可

KV场景

RocksDB

RocksDB采用不跨页的方式，在组KV时，通过BlockBuilder实现，如果当前Block不能容纳新的KV，同样换一个新页，在空白部分填充padding

想法

Block的组织可以分为两种，一是数据块紧密排布，这样最节省空间，但是数据块容易出现跨页的问题这里称作tight模式，二是如果page装不下当前数据块，则分配新页，保证数据块不会跨页（超过page的大数据块则不会额外增加一次跨页）这里称为strict模式
当我们想要修改数据块的组织方式时，两种模式的优化目标有所不同，tight模式则会尽可能减少跨页，而strict模式则是尽可能减少空洞的出现。

优化两种组织

毫无疑问，对block的重新组织，至少要遍历一次，我们分别称为TightSort和StrictSort。

strict优化

对于StrictSort可以抽象为装箱问题，因为我们主要关注的是不足page的部分

解法参考：https://drwxyh.github.io/2018/08/15/Bin-Packing/

可以用First-Fit-Decreasing Algorithm，时间复杂度在比最优的比例大概在也就是会多用0.22的空间

  • 先从大到小排序（根据%page_size）需要
  • 然后找一个当前能装下的page，找不到就新分配一个 page可以 用一个平衡二叉树实现

tight优化

对于一个blcok序列，我们期望在任意的序列中等于0的个数近可能多
考虑用贪心的方式，在每次组装序列中，选择和能够使当前和尽量逼近0

  • 先对block进行排序数组
  • 当sum%page_size不等于0时，则查找page_size-sum对应的block，否则拿一个最小值 O(1) 用哈希表实现;
  • 贪心之处在于小block先装进去，使得block减少跨页。

思考

存储一般是便宜的，计算和时延是昂贵的，优化虽然简单节省了空间和跨页，但是会对写入造成较高延时，采用Strict的方式应该是最佳实践。
也就是最多只能支持的时间复杂度，甚至还需要考虑常数单位
