对bRPC的IOBuf理解

zyu Lv2
  2025-08-30 2025-08-30 Created   2025-10-18 09:59:42 2025-10-18 09:59:42 Updated  
  • Tech 
  • | Code 
  • | RPC 
 

bRPC的高性能，来源于一些其独到的设计，其文档已经相当全面，这里主要是根据其实现学习一下，谈谈自己的理解。

协议

通信双方，重要的是协议，才能互通，其中开头的字节说明了其协议类型

相比HTTP2协议，百度用了baidu_std的协议来提升性能，主要优化点在于

  • 协议头（Protocol Header）非常紧凑，只包含必要的元数据（如 magic_number, body_size, meta_size），减少数据传输量
  • 使用IOBuf进行零拷贝
  • 流式处理
  • 使用Attachment，二进制格式的非结构化数据，这使得比如说图片这些二进制数据，或者直接转发非常方便（配合零拷贝）
    baidu_std的结构为，其中body为protobuf body是结构化数据
HeaderMetaPayload
protocol(4B) + body_size(4B) + meta_size(4B)控制信息，其中包含attatchment_sizebody + attachment

协议的优势在于分离做得很好，

  • RPC 的元信息（Controller 中的设置，如 trace id, 超时时间等）和业务数据（用户定义的 Request/Response 消息）在逻辑上和物理上分离。元信息序列化后存储在 Meta 部分，业务数据序列化后存储在 Body 部分
  • 序列化信息和非序列化信息也做了分离，这使得可以只解析body，由于attachment支持零拷贝，因此可以把它当作一个对象即可，方便转发

关于Attachment
Attachment不会自动填充到 Protobuf response 对象里，在回调中调用 cntl.response_attachment()，返回一个 butil::IOBuf。自己来解析这个 IOBuf，或者将它传递给后续逻辑。

IOBuf

参考：https://brpc.apache.org/zh/docs/blogs/sourcecodes/iobuf/

当然，IOBuf不是只有baidu_std在使用，本质上只是高效的数据结构，IOBuf核心在于引用，通过对引用的操作来绕开对数据的直接操作，它的接口和std::string类似，那不可避免地要引入一个中间逻辑层 即BlockRef

整体结构上来说，首先内存被Block的形式进行划分，很通用的设计思路，每个申请的内存区域是以BlockRef进行管理的，而IOBuf就是BlockRef的集合，因此数据块的拼接等操作都是移动BlockRef实现的

Block

如果要改成RDMA场景，则修改Block的分配来源即可

BlockRef

作为Block内某块区域的管理，那我们三个元数据信息就可以了，所在块和具体位置信息

IOBuf

IOBuf作为BlockRef的集合，设置两种方式BigView和SmallView的联合体，设计倒是跟folly很像

  • SmallView 内部直接包含一个固定大小的数组（refs[2]），这个数组和 SmallView 结构体本身都存储在 栈 (Stack) 上。
  • BigView 内部包含一个指针 (refs*)，这个指针指向一块在 堆 (Heap) 上 动态分配的内存，这块内存里存放着 BlockRef 数组。

零拷贝

有了Ref这层引用，那在一些接口上实现零拷贝，就比较容易了,直接把Ref添加到IOBuf的数组中

其他

Block竞争问题

如果全局的Block池子，创建和回收Block都会有很高的并发控制开销，这部分通过TLS来解决

  • 有一个GlobalPool和线程独有的LocalPool
  • 在create_block()时该机制被使用
    对于小对象，则会从 TLSData 获取同一个 Block 来写入
    bthread是用户态协程，如果用pthread 的 TLS， 当一个物理线程从 bthread A 切换到 bthread B 时，它们会共享同一个 TLS 数据，达不到bthread隔离的要求，每个bthread都关联一个TLSData

想法

和string_view的冲突
在string等大对象的放进pb 对象的时候，是需要一次拷贝的（业务逻辑），因此有优化是用string_view来减少拷贝，但是baidu_std下 client 端好像拿不到解析好的 pb response，因此得用HTTP2协议
