方法论

性能遇到瓶颈的时候，比如调用时延高，又或者服务可用性下降，我们应该怎么分析？
在《性能之巅》中设计了对应的模型和方法论，模型是将整个系统建立为受测系统和排队系统。受测系统即接受输入，产生性能结果，关键之处在于受测系统会受到外部因素的扰动。排队系统是重要的理论抽象，输入请求到达，如果服务中心繁忙，请求就会在队列中等待 。处理完成后，变为输出，是进行性能分析，资源分析的基础
方法论为：

  • 问题：首先清晰地陈述性能问题
  • 假设：基于问题，假设性能不佳的可能原因
  • 预测
  • 试验
  • 分析

问题往往是直观的，但不是直接的，比如调用时延高是直观的，但或许不是性能问题，因为不是性能问题，比如CPU占用率高即可作为表层原因，也可以作为性能问题。再比如调用成功率下降可能不是性能问题，而是触发服务拒绝策略了。
因此遇到第一个问题时，我认为先进行问题的明确化，所谓定位，更准确地说是缩小范围，从受测系统的角度出发，突发情况中，往往是有多个异常值，通过异常值实际上能够缩小受测系统的范围，比如不需要从整个服务上下游出发，而是定位到某个服务中。服务，程序毕竟是按流程执行的，这个时候我们可以按照流程进行定位，比如服务中可以调用各个下游服务，程序运行的各个部分。
其实这个异常值也代表了受测系统的一个重要部分，观测值，系统能观测到哪些部分，则会让我们的定位更加精确。
当定位到一个受测系统，比如某个服务器上的服务，可以进行两部分分析，外部因素和资源分析（排队系统）。这里的外部因素，可以简单归类为负载，（当然其他也有，比如网络环境大流量的干扰呀等等，但比较难观测），《性能之巅》描述为工作负载特征总结。
接下来做资源分析，《性能之巅》提出了快速方法USE方法，和全栈的系统分析方法

USE方法

对于所有的资源，查看它的使用率、饱和度和错误（优先检查错误）

  • U - 使用率 (Utilization)：资源用于服务工作的时间百分比 。对于存储等基于容量的资源，它指的是所消耗的容量（例如，内存使用率） 。
  • S - 饱和度 (Saturation)：资源无法再服务更多额外工作的程度，通常表现为有工作在排队等待 。
  • E - 错误 (Errors)：错误事件的个数

全栈系统分析

理解所有组件的关系，知道组件哪里出现了异常，这需要

  • 系统基准评估，比如SSD支持的最大带宽是多少等，这些作为基准，能够帮助我们快速定位异常值
  • 系统资源瓶颈评估
  • 程序根因定位：虽然我们知道瓶颈出在哪，但是一个服务程序可能卡在某个线程，又或者说哪里拖慢了我们的程序，这个时候需要进行相应的定位，定位到某个代码

perf

perf 用户态程序通过 perf_event_open 系统调用，请求内核配置 CPU 的硬件性能监控单元 (PMU) 或内核中的软件追踪点 (Tracepoints)，来收集性能事件

perf 可以监控的事件来源主要分为三类：

  1. 硬件事件 (Hardware Events)
    • 来源： CPU 的 PMU (Performance Monitoring Unit)。
    • 原理： 现代 CPU 内部都集成了 PMU，它包含一组特殊的计数器 (PMC, Performance Monitoring Counters)。这些计数器可以被编程来统计各种底层硬件事件，例如执行了多少条指令、发生了多少次 L1 缓存未命中、有多少次分支预测失败等。
    • 优点： 性能开销极低，数据非常精确，因为这是硬件自己在数数。
  2. 软件事件 (Software Events)
    • 来源： Linux 内核。
    • 原理： 内核自身在运行过程中也会产生很多有意义的事件。内核在这些地方预先定义好了“钩子”，当事件发生时，内核会主动增加计数器。
    • 示例： context-switches（上下文切换）、page-faults（缺页中断）、cpu-migrations（CPU 迁移）等。
  3. Tracepoints (追踪点)
    • 来源： Linux 内核中静态插入的“探针”。
    • 原理： 内核开发者在代码中（例如系统调用的入口/出口、调度器关键路径）手动插入了 TRACE_EVENT(...) 宏。perf 可以“挂载”到这些点上，当代码执行到这里时，内核就会通知 perf。
    • 优点： 可以获取非常详细的内核运行信息，例如某个系统调用被谁调用了、参数是什么。
      此外，还有 kprobes（动态内核探针）和 uprobes（动态用户态探针），它们允许 perf 动态地在任意内核函数或用户态函数中插入探针，实现更灵活的追踪
