来自FAST’25的会议论文，GPU的显存容量有限，因此需要把数据放到外部的持久设备中，特别是PM，问题在于访问这些设备需要CPU来处理数据访问逻辑，增加了CPU和GPU的同步开销，已经有机制和工作允许GPU访问块存储设备，但是和PM的粒度仍然不一致，论文为此提出了一个高效的哈希索引。

背景

为什么需要持久设备
比如大量embedding数据（可能会来到TB级），这个时候内存肯定存不下。因此放到持久设备中，但是需要CPU的管理
CPU参与有什么问题
如下图蓝色部分，增加了数据的拷贝和CPU的额外消耗

GPUDirect
GPU显存和SSD之间提供块级别的访问接口，即GPUDirect，通过DMA直接传输数据。但是不适合以字节粒度的索引，GPU 与持久内存结合（GPM，之前已经有工作实现了）允许 GPU 通过内存总线直接以字节粒度访问 PM，如上图红线部分

  • 通过CUDA的UVA统一虚拟地址编码把PM映射到GPU的虚拟地址
  • 使用系统级（System Scope）的内存屏障指令threadfence_system()把缓存在 GPU L1/L2 Cache 或 Write Buffer 中刷写出去，使得对系统可见
  • 关闭CPU的DDIO，因为DDIO默认开启， 外设（如 GPU）通过 PCIe 写入数据到主机内存时，数据会优先写入 CPU 的 L3 Cache (LLC)，而不是直接写入物理内存（DRAM/PM）
    GPU和哈希索引
    哈希的点查性能很高，很适合一些负载，比如查找embedding

挑战

不能直接把Hash移植到GPU上，因为

warp执行特性

传统索引忽略了 GPU 的 Warp（线程束）执行特性，导致严重的 Warp divergence（分歧）和非合并内存访问（Memory Coalescing），性能低下

warp divergence

GPU为SIMT（单指令，多线程）架构
一个 Warp（通常包含 32 个线程）中的线程在执行条件跳转（如 if-else）时，就会发生 Warp Divergence，一个 Warp 里的 32 个线程共享同一个 程序计数器（PC）, 只有一个取指单元。因此在于同一时刻，只能执行一个指令。
此时GPU会自动进行串行化，

  • 第一步：激活满足 if 条件的线程（Active Mask 设为有效），禁用不满足的线程（Mask 设为无效），执行 if 块中的指令。
  • 第二步：反转掩码，激活走 else 分支的线程，禁用刚才走 if 的线程，执行 else 块中的指令。
  • 第三步：所有线程在分支结束处（Re-convergence point）重新汇合，继续并行执行。
    因此warp divergence的问题在于破坏了GPU的并发性

Memory Coalescing

如果Wrap中 32 个线程请求的内存地址是连续的（位于同一个缓存行 Cache Line 中），GPU 硬件会将这 32 次请求合并成 1 次 物理内存事务。

哈希索引的不适配

  1. uncoalesced memory accesses：由于哈希索引的特性，warp每个线程访存地址会不同，此时访存就会变为32个请求，占用大量内存总线带宽
  2. warp divergence: 不同线程在哈希索引中，比如链式哈希可能要跳转的次数不同，导致了需要等待最慢的线程
  3. 竞争：哈希索引如果用锁进行管理，那么对GPU这么多线程，会造成强烈的竞争

崩溃一致性

PM的崩溃一致性有最大粒度，来自内存总线宽度（64位）限制，通过执行
flush and fence 指令实现，之前的解决方案logging或者COW都会造成额外的写。

  • 日志：先写日志（旧数据写日志是undo, 新数据写日志是redo）然后原地更新
  • COW: 更新指针

带宽差异

PM访问延迟是DRAM的3-10倍，带宽是DRAM的1/6，V100可以到900GB/s，而6个Optane DIMM读带宽只有39.6GB/s，写带宽只有13.8GB/s

哈希冲突

核心是level hash的设计

level hash适合GPU的特性

  • 一个bucket有多个slot，对于GPU线程可以并行一次取
  • GPHash进一步扩展了level 的层数 一次hash映射可以指向每个level 提升了内存访问效率，GPU并行可以取所有level的slot
    其他参数
  • 每层的bucket数目 = $2^{level}$
  • 上层共享下层的bucket
    GPHash做的设计
- 原地更新，而不是指针，来避免scatter的内存访问 - 对于value还是用的指针，并且CAS更新 ### 并发控制 和 崩溃一致性 经典的无锁化设计 - CAS控制标志位，和level hash基本一致 - PM崩溃一致性这个点的优化跟level hash基本一致

标志位：empty->insert->key的hash_value(类似commit)
