背景以及问题

存储系统基于SEDA的设计原则，(Staged Event-Driven Architecture) 处理请求的时候会分阶段进行，每个阶段用事件队列通信

也就是a thread-per-stage (TPS) architecture ，其中每个阶段可以用线程池来处理，比如RPC 请求处理，同一阶段基于操作系统的调度来并发处理任务
在高性能硬件中，比如RDMA DPDK等调度比硬件处理开销比较大，所以会采用不可抢占的绑核，很多内存KV使用了这种模型(RTC, run to completetion)，一个线程执行完所有的阶段，这样带来问题：会遭遇严重的缓存效率低下和阻塞问题

  • 不同阶段负载特征不同，进入不同阶段，造成缓存抖动问题
  • 单线程中串行执行不同阶段，容易造成阻塞

线程模型分类

论文对线程模型做了分类

  • 抢占式架构：允许CPU在等待I/O时切换到其他任务
    • P-TPR (Thread per request，每个请求一个线程)： 会导致线程激增
    • P-TPQ (Thread per queue，每个队列一个线程)：系统启动时创建一定数量的线程，它们循环地从各自的专用队列中获取并处理请求
    • P-TPS (Thread per stage，每个阶段一个线程)： 模块化和性能隔离
  • 非抢占式：
    • NP-TPR： 为避免创建和销毁OS线程的高昂开销，这类系统使用“绿色线程”或“用户态线程”
    • NP-TPQ：
      • 绑核： 减少调度开销
      • 批量处理：减少和硬件交互开销
      • 分片：每个线程数据分片
      • 协程：调度，减少CPU空转
    • NP-TPS: 原因是传统的 TPS 模式会引入频繁的跨阶段、跨线程通信，这被认为不利于充分发挥高速硬件的性能
      要把TPS的模型用在这些RTC模型上，面临有
  • 每个阶段要留多少资源？
  • 不同阶段的通信开销
    总而言之，要解决的是通信和调度
    此外NP-TPS的设计空间包括
    1. 缓存命中率，分离不同阶段可以提升缓存命中率
      1. 网络包可以直接进入三级缓存
    2. 热点的资源隔离，可以保证LLC等不被污染
      1. 有CAT这种划分LLC的
      2. DDIO 通常通常使用LLC中最右边的两个路， 是否可以通过阻止工作线程使用为DCA保留的缓存路来缓解NP-TPQ的缓存效率低下问题， 实际上DDIO仅在缓存未命中触发的缓存分配时才使用LLC中最右边的两个路；如果缓存行已经驻留在LLC中（超出DCA保留的路），DDIO会直接修改或访问它。因此仍然 有缓存干扰
    3. 具有通过调度解决倾斜的负载

方案

CR 层

作为缓存层，基于

  1. LLC 缓存现在已经很大了 甚至有504MB(我的开发机也有27.5MB)

设计点

  1. 尽量让热点数据(KV对)在LLC中，网络的解析/响应放在LLC中，这个点我认为应该基于DDIO的设计考虑，跟热点的关系不是那么大
  2. 专属核心和LLC来保证缓存的稳定性
    LLC现在的控制不是直接的，怎么把热的KV对放进LLC中的？
    采用一个后台线程，维护一个hot set 在内存中,
  • 后台线程定期采样最近访问的键
  • 使用count-min sketch 和最小堆的组合来跟踪最热的数据项（实现中为10K个数据项）
  • 基于epoch的方法切换set
    CPU在访问时会自动将它们加载到缓存中

MR 层

减少访存时间

  • 利用C++的无栈协程，同时查找多个key, 每个key查找对应一个协程，然后设置prefetch和co_yield在指针解引用的时候 即访存时，切换协程，因为协程切换只要几个纳秒
    • C++20的协程是编译器上的无栈协程
      减少数据拷贝和通信开销
      直接在网络缓冲区和 KVS 存储位置之间进行数据拷贝
      用一个消息队列批量取消息

自动调参

参数：1.thread num: 线程数量 2.线程LLC大小 3. cache size: CR缓存的hot item
约束：

  1. cache size调整会影响负载，进而需要调节thread num
    思路：三分搜索
    调节thread_num 和 LLC 都会出现系统性能先增后减的凸函数
  • 如果 $f(b) > f(a)$ ： 那么值在$[a, max]$中搜索
  • 如果 $f(b) <= f(a)$ : 那么值在$[0,b]$中搜索

思考

论文最重要的是提出的问题，论文“批评”的那些项目是学术界开源设计的，相比工业界项目肯定是设计很粗糙，性能比较差，毕竟核心是印证性能，论文提到的大部分TPS项目其实都是工业界的，论文填补的空白是学术界的开源项目设计上的不足? : )
珍贵的点是总结（炒冷饭）了系统架构设计思路，同时有理有据（基于数据负载的特性来划分阶段，即缓存友好性），是非常不错的启发论文，适合作为架构师的扩展读物。
但是有几个点我觉得论文没解释清楚

  1. 请求处理到KV其实要走很多流程，比如解析，每一步都会刷LLC，这个时候真的体现LLC的优势吗
  2. 关于用协程掩盖指针追逐延迟，随便写的demo，协程相比单进程能实现10倍的提升，应该是成立，但是这个点或许不算特别的设计点
