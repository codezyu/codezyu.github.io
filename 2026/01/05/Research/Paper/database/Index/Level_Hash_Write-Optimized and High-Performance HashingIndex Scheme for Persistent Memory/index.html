论文来自OSDI’18 NVM 存在两个主要短板：写入寿命有限（endurance limited）以及写入性能低于读取性能

问题

  1. 数据一致性与写入开销一致性开销大：为了保证断电后的数据一致性，必须通过 CPU 指令（如 CLFLUSH, MFENCE）强制将数据刷入 NVM，这带来了显著的性能开销。此外，由于处理器原子写通常不超过 8 字节，写入大数据需要昂贵的日志（logging）或写时复制（CoW）机制。
  2. 传统哈希的不适应：传统的哈希方案（如链式哈希、Cuckoo哈希）在解决冲突时会产生额外的指针修改或级联写入（cascading writes），这加剧了 NVM 的磨损并降低了性能。
    现有NVM方案的缺陷：现有的 NVM 友好型方案（如 PFHT, Path Hashing）虽然减少了写入，但往往牺牲了查询性能或负载因子（Load Factor）。
    Resizing（扩容）代价高昂：传统的哈希表扩容需要 $O(N)$ 的时间复杂度和大量的写入操作，严重影响系统性能。
# 解决方案

哈希冲突

对于哈希来说，写入的最大开销来源于哈希冲突，level hash采用两个设计点，两层的结构和两个哈希函数。
采用两层结构（Top Level 和 Bottom Level），底层作为顶层的备用区域，数量为顶层的一半。

  • 底层用来解决顶层的哈希冲突问题，顶层两个桶共享底层的一个桶 对应设计点D3
  • 每个桶包含多个slot -> slot的设计点在于很多kv对都是小value，对应设计点D1
    通过两个哈希函数定位，使得查询、插入、删除在最坏情况下仅需探测常数个（最多4个）桶，实现了 $O(1)$ 的时间复杂度 对应设计点D2

    我觉得解决冲突的核心点，本质还是解决 不均匀的问题，相比于链式哈希，level hash的核心是用来共享的桶，使得相邻的可能实现互补，提高了利用率
    缺陷肯定是要扫描更多bucket，对于写优化来说是合理的

同时为了限制比如Cukoo hash这种级联踢出元素带来的一系列写问题，level hash则是最多移动一个元素（移动到另外一个top 或者 bottom），如果新插入的上下两层都满了就直接扩容，考虑的点在于NVM不适合太多的写 对应设计点D4

迁移的过程本质就是为了均匀数据分布，大部分来说，我感觉一次数据移动就够了

最大的提升还是来自D2 和 D3 两个提升了数据分布均匀

resize

resize的缺陷在于需要进行重哈希大量写入，level hash 它不移动所有数据，而是通过调整层级关系，只移动 $1/3$ 的数据。

初始时刻 TL: N BL: $N/2$

  1. 系统分配一个新的内存区域，包含 $2N$ 个桶，作为新的顶层
  2. 原来的旧顶层 (Old TL) 直接变为新的底层 (New BL)。注意，这里是逻辑角色的转变，数据不需要移动，物理位置不变
  3. 原来的BL进行重哈希，到新的TL上

    很有思路的结构设计，说明了映射关系是逻辑映射的重要性，这样更容易调整结构

Bottom-to-Top (B2T) Movement（自底向上移动）：

  • 为了解决底层过满导致的负载因子下降问题，当新插入数据发现对应位置满时，Level Hashing 会尝试将底层的数据向上移动到顶层的空位中，从而重新平衡上下层的负载
    Dynamic Search（动态搜索）：
  • 由于扩容后大部分数据实际上位于底层（旧顶层），传统的“先查顶层、再查底层”策略会导致搜索效率下降。
  • 如果底层的元素数量多于顶层，搜索操作将优先探测底层。这显著降低了扩容后的平均搜索延迟

一致性保证

Level Hashing 实现低开销一致性保证的核心在于：利用小于 CPU 原子写粒度（8字节）的元数据（Token）作为”生效开关”，从而避免对数据本身进行昂贵的日志记录。
具体实现方式总结如下：

  1. 依托原子 Token： 每个桶头部维护一组 Token（每个槽位 1 bit），其总大小（通常 1-2 字节）允许通过单条 CPU 指令进行原子更新。
  2. 无日志插入 (Log-free Insertion)： 先将数据写入 NVM，利用内存屏障（MFENCE）确保持久化，最后原子修改 Token 从 0 变为 1。若中途断电，Token 仍为 0，未完全写入的数据被视为无效，保证了一致性。
  3. 无日志删除 (Log-free Deletion)： 直接原子修改 Token 从 1 变为 0 即可使数据失效。
  4. 机会性无日志更新 (Opportunistic Log-free Update)： 更新时若桶内有空槽，直接将新数据写入空槽，然后在一次原子写操作中同时翻转两个 Token（将旧数据的 Token 设为 0，新数据的设为 1）。只有在桶全满这种极少数情况下，才退化为使用日志。
  5. 容忍临时重复 (Duplicate Tolerance)： 在扩容或移动元素时，遵循”先开启新位置 Token，后关闭旧位置 Token”的顺序。系统设计允许在崩溃恢复后存在暂时的数据重复（读取任意一个均正确），从而避免了复杂的撤销（Undo）日志。

    这个核心的设计点是，把日志和原来的数据位置设置为一个，如果是insert/delete，那么是合理的，对于update，还是需要额外的空位，原子性保障这里，简直太厉害了，非常好的一篇工作

总结

非常好的工作，值得细细研究
